## Update — Track A (streaming), 2026-07-07

BUG-001: **fixed**, commit `92cebda` on `track/streaming` (SSE-extraction prep commit `a61f6ad` precedes it).

Root cause was broader than the repro's literal fetch list suggested: both "New session" entry points (`src/routes/+page.svelte`'s hero button and `src/lib/components/command-palette.svelte`'s ⌘K item) navigate straight to `/sessions/{key}` with no `?start=`, which put `+page.svelte`'s `afterNavigate` in the "existing session" branch — the one that unconditionally fires `/approvals`, `/transcript`, and `/runs`. Since sessions are created lazily (the DB row is only written inside the workflow on the first `POST /turn`), all three requests raced the not-yet-created row.

Fix: "New session" now navigates with `?fresh=1`. `src/lib/session-page-bootstrap.ts` is the single decision point for what the session page fetches on mount — a fresh, message-less session fetches nothing (there's nothing to fetch yet); a fresh session with `?start=` submits that first turn (which is what creates the row); an existing-session revisit loads pending approval, transcript, and run inspector as before. Regression test in `src/lib/session-page-bootstrap.test.ts` asserts the actual call ordering, not just a boolean.

  | SEC-002 | high | CSRF / cross-origin request forgery on every mutating `/api/*` endpoint (`POST /api/approvals/[approvalId]/resolve`, `POST /api/sessions/[sessionKey]/turn`, `/steer`, `/interrupt`, `PATCH /api/sessions/[sessionKey]`, `POST /api/schedules`, `POST/DELETE /api/schedules/[scheduleId]` and its `/pause`, `/resume`, `/trigger` actions, `POST/DELETE /api/sessions/[sessionKey]/memory/candidates/[candidateId]`). None of these endpoints, nor any global hook, validated the request's `Origin`. SvelteKit's own built-in CSRF check (`csrf_check_origin` in `@sveltejs/kit/src/runtime/server/respond.js`) is skipped entirely when `!DEV` is false — i.e. it is a no-op under `bun run dev`, which is how this app is normally run — and even in production it only inspects requests whose `content-type` is one of the three HTML-form content types, never `application/json`. Since these handlers all parse `request.json()` regardless of the declared `content-type` header, a malicious webpage open in another tab while `bun run dev` is running could fire a same-site-simple `fetch(url, { method: 'POST', headers: { 'content-type': 'text/plain' }, body: JSON.stringify(...) })` at `localhost` and have it silently accepted — e.g. auto-approving a pending `shell.exec` approval, submitting an arbitrary turn, or deleting a schedule. This app has no other auth layer (it's a local, single-user dev tool), so origin was the only available boundary check and it was missing everywhere. Fixed by adding `src/hooks.server.ts`, a single `handle` hook that rejects any non-GET/HEAD/OPTIONS request to `/api/*` whose `Origin` header doesn't match the request's own origin (including requests with no `Origin` header at all, which are treated as untrusted since this endpoint set has no other identity check). Active in both dev and prod, unlike SvelteKit's own check.
